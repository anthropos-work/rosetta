# M10 — Decisions

_Implementation decisions with rationale. ID scheme: M10-D1, M10-D2, …_

| ID | Decision | Rationale | Date |
|----|----------|-----------|------|
| M10-D1 | **Generalize the firewall to a per-surface `PublicPredicate`** (scope columns + public SQL filter + label) instead of the hardcoded `organization_id IS NULL`. Taxonomy/reference keep the org-only `DefaultPredicate` (zero-value falls back to it → byte-for-byte unchanged); `directus` declares `private=false AND tenant_id IS NULL AND status='published'`. Threaded through `AssertPlan`/`FilterFor`/`ParentScopeFilter`/`BuildPlan`/`CopyPublic`-leak-probe + recorded in the manifest's optional `predicate` label. | The spike-flagged architectural gap: the framework was org-only. A per-surface predicate is the minimal generalization that lets directus reuse ALL of the plan/capture/parent-scope/firewall machinery. The post-capture scope-bearing leak probe became `count(*) WHERE NOT (<public filter>)` (was `<org> IS NOT NULL`) — same 0-by-construction guarantee, predicate-agnostic. | 2026-06-06 |
| M10-D3 | **Directus public predicate = `private = false AND tenant_id IS NULL AND status = 'published'`** (user-locked Decision 3). The directus surface is 9 tables in FK order: `simulations` + `skill_paths` (scope-bearing roots) → `resource` (pure-reference global library) → `roles`/`sim_tasks`/`sequences` (parent-scoped via simulations) → `task_checks`/`task_sub_checks` (MULTI-LEVEL chains) → `sequences_roles` (via BOTH sequences AND roles). | Prod-verified counts (read-only, 2026-06-06): 304 sims / 22 paths / 1,543 resources / 953 roles / 949 sim_tasks / 304 sequences / 2,242 task_checks / 2,850 task_sub_checks / 953 sequences_roles. Strict 22 paths (not the ~190 estimate — that omitted `tenant_id IS NULL`). Directus has NO DB-level FKs (relations live in the app layer), so closure is by convention, verified via the parent-scope counts. | 2026-06-06 |
| M10-D4 | **Multi-level parent chains via `ParentScope.ParentFilter`.** Directus children like `task_checks` parent a COLUMN-LESS intermediate (`sim_tasks`, no private/tenant_id) that is itself parent-scoped. Rather than nest ParentScope structs, a ParentScope carries an optional `ParentFilter` (the inner WHERE selecting the parent's public id set) that chases to the scope-bearing root in one subquery: `task IN (SELECT id FROM directus.sim_tasks WHERE simulation IN (SELECT id FROM directus.simulations WHERE <public>))`. Threaded through `ParentScopeFilter` + the post-capture leak probe. | The taxonomy surface only had scope-bearing parents (skills/job_roles carry org columns). Directus's deeper template tree needs the column-less-intermediate case. ParentFilter is the minimal, audit-legible extension; an empty ParentFilter keeps the scope-bearing-parent default (taxonomy unchanged). | 2026-06-06 |
| M10-D2 | **Store fork = per-stack Directus container on the per-stack Directus-backing Postgres** (user-locked Decision 1). Replay the captured public content into each stack's OWN Postgres `directus` schema via the framework's `COPY` path (class `postgres`=`PerStackIsolated`=always-allowed), then boot a per-stack Directus pointed at it. Provision the `directus_*` system schema via Directus's own first-boot migrations (schema structure), then replay the content rows. NOT fed via the Directus write/import API (slow, schema-fragile, a write path the guard would have to trust). | The spike recommended this; the user locked it. Directus stores all content in Postgres → its backing DB is per-stack-isolated by construction, so the existing bulk-`COPY` replay applies cleanly with no special guard exception. The shared prod Directus (`SharedPollutionRisk`) is never written — capture is read-only via `marco_read`. | 2026-06-06 |
| M10-D5 | **S3 media blob BYTES → Fate 2 → v1.3 (confirmed-covered, no roadmap edit).** The media-REFS path (ref columns, `ReferencedFilesFilter`, the 1,311 public `directus_files` rows, the per-stack local-storage + placeholder adapter) is the milestone FLOOR and **landed**. The blob BYTES (read Directus's own S3 → write a per-stack-isolated bucket) are gated on confirmed S3 read access not wired in this environment → `BlobBytesAvailable()` is the honest, tested `false` gate. Route the bytes to the **existing** v1.3 "Cloud snapshot store / S3 backend" seed in `roadmap-vision.md` (user note #4, 2026-06-06). | Fate-1 is infeasible without provisioning S3 credentials (out of M10's content-snapshot scope); a "stub a byte-copier" would be a disguised deferral the three-fate rule rejects. The v1.3 home already exists (same S3-access-gated infra class) — no new roadmap entry invented at close, NOT escape-hatch (the structural floor delivers M10's 100%-coverage thesis; v1.2 scope is not broken). Recorded at the close per the Phase 1b deferral re-audit (GREEN). | 2026-06-06 |

## KB-fidelity findings (Phase 0b — track to Phase 5)
- **KB-1** — `db-access.md` + `snapshot-spec.md` describe the public Directus content as a "separate self-hosted
  Directus store / its own Postgres" with no reachable DSN. DISPROVEN by locked Decision 2: it is a `directus`
  schema inside the SAME `postgres` DB, read-only via `marco_read` MCP (verified `directus.simulations`=2,597).
  Fix in Phase 5 docs (M10 Delivers).
- **KB-2** — the snapshot firewall predicate is hardcoded `organization_id IS NULL`. M10 generalizes it to a
  per-surface public predicate; the `directus` surface uses `private=false AND tenant_id IS NULL AND
  status='published'`. Document the generalization in Phase 5 (snapshot-spec.md).
