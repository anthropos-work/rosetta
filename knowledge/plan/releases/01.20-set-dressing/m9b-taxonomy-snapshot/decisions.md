# M9b — Decisions

_Implementation decisions with rationale. ID scheme: M9b-D1, M9b-D2, …_

| ID | Decision | Rationale | Date |
|----|----------|-----------|------|
| M9b-D1 | The `taxonomy` surface is **10 tables**, captured in FK (replay) order: `categories`, `job_role_categories` → `specializations` → `skills`, `job_roles` → `skill_embeddings`, `job_role_embeddings`, `skill_translations`, `job_role_translations`, `job_role_skills`. | Prod FK graph (pg_constraint, 2026-06-06): `skills.parent→specializations`, `specializations.parent→categories`, `job_roles.category_id→job_role_categories` (a SEPARATE table, NOT `skiller.categories`), embeddings/translations→their parent, `job_role_skills→{job_roles,skills}`. Referential closure requires every referenced parent in the capture set, so `job_role_categories` must be added (it was not in the overview's table list — surfaced during build, landed Fate 1). | 2026-06-06 |
| M9b-D2 | **Parent-scoped capture filter** — column-less tables (embeddings/translations/job_role_skills) are captured with a real subquery filter `WHERE <fk> IN (SELECT id FROM <parent> WHERE organization_id IS NULL)`, AND the post-capture tenant probe re-checks via the parent join. The M9a framework recorded `PublicVia` in the manifest but applied an EMPTY capture filter to column-less tables (it would have captured the whole table incl. customer rows). M9b closes this: `TableSpec.PublicViaFK` + the firewall/adapter build the parent-scoped predicate. | The load-bearing read-side safety for the taxonomy. Without it, `skill_embeddings`/etc. would leak customer-parented rows. M9a only proved the toy `item_vectors` case where the whole column-less table happened to be public; the real surface has customer-parented embeddings. | 2026-06-06 |
| M9b-D3 | **`job_role_skills` is scoped by BOTH endpoints public** (role AND skill), not the role alone. | Q3 RESOLVED: prod has **3** rows where a PUBLIC role links to a CUSTOMER skill (`72559` public-role links − `3` = `72556` both-public). Scoping by role alone would capture 3 links whose `skill_id` is a customer skill absent from the public capture set → FK orphans on replay. Both-endpoints scoping keeps the surface referentially closed. | 2026-06-06 |
| M9b-D4 | **Single streamed `COPY`** per table (no keyset chunking). | Q1 RESOLVED: the public surface is ~2.1 GB but the heavy weight is the `skill_embeddings` INDEX (~689 MB), which is NOT transported (rebuilt on replay). Heap payloads stream within the 30-min `statement_timeout` bound. Keyset chunking would add complexity with no benefit at this size; revisit only if a future surface exceeds the read bound. | 2026-06-06 |
| M9b-D5 | **pgvector index rebuilt via `REINDEX TABLE`** on replay; the vector column + dimension (1536) recorded in the manifest, gated by the embedding-dimension gene. Index TYPE/params (IVFFLAT per skiller.md) are NOT replicated verbatim — `REINDEX` rebuilds whatever index DDL the stack's migrated schema already defines. | Q2 RESOLVED: the replay target is a FRESH migrated stack whose `skill_embeddings`/`job_role_embeddings` already carry the IVFFLAT index DDL from the platform migration (`20260417*`). Replay loads the rows then `REINDEX`es to populate that index — we don't re-issue `CREATE INDEX` (the migration owns the params). The manifest records `vector_columns` + dim so the fidelity gene confirms the replayed vectors carry width 1536. | 2026-06-06 |

| M9b-D6 | **The snapshot-fidelity measure is wired via a manifest→CapturedTable bridge + a `datadna measure-snapshot` CLI**, not a cross-module import. `dna.CapturedFromManifest` parses the snapshot `manifest.json` (a minimal local struct mirror; refuses a non-public-only manifest) into the source-side expectations; `PgFidelityProbe` reads the replay side; `MeasureSnapshot` composes the score and gates on critical 100%. | M9a left `RunSnapshotOperators` taking `[]CapturedTable` with NOTHING constructing them from a real capture (the tests hand-built them). This was the load-bearing measure gap. The two modules are separate (stack-seeding vs stack-snapshot), so a JSON-via-minimal-struct bridge (with `manifestFormatVersion` kept in lockstep) avoids a hard cross-module dependency while reading the real manifest. | 2026-06-06 |
| M9b-D7 | **The taxonomy snapshot is a DAG node (`TaxonomySnapshotSeeder`), not a row-by-row seeder.** It declares `Surface()=="taxonomy"`, depends on nothing (taxonomy is independent public reference data), and its `Seed()` VERIFIES the snapshot was replayed (counts public `skiller.skills`) rather than writing rows. `activity` now `DependsOn("taxonomy")`. | The actual data load is a privileged-read→per-stack-write done by the separate `stack-snapshot` extension (`stacksnap replay`) BEFORE `stackseed` — a different blast radius from a generated-row seeder. Making it a real `Seeder` that COPYs rows would be a leaky abstraction. The verification node makes the DAG ordering (`… → taxonomy (snapshot) → activity`) real + tested while keeping the load in its own extension. | 2026-06-06 |
| M9b-D8 | **The generated `ts_search` tsvector is EXCLUDED from every captured column set.** | It is regenerated by the stack on load (a derived column); transporting it would be stale on replay and pointless. The captured set is identity + descriptive + scope columns. | 2026-06-06 |

## Open at design — RESOLVED during build
- ~~M9b-Q1~~ → **M9b-D4** (single streamed COPY).
- ~~M9b-Q2~~ → **M9b-D5** (REINDEX TABLE; migration owns index params; dim 1536 in manifest).
- ~~M9b-Q3~~ → **M9b-D3** (both endpoints public — 3 customer-skill links excluded).

## Review fix (PR-review Phase 3/4)
- **Parent-scope leak probe must AND the capture filter.** First cut counted customer-parented rows across the WHOLE live table → would FALSELY ABORT a correct capture (skill_embeddings has customer-parented rows the filter correctly excludes). Fixed: `buildParentLeakProbe` ANDs the capture filter (count leaks WITHIN the captured set, 0 by construction), extracted as a pure function + unit-tested. Extensions commit `0404760`.

## Prod evidence (2026-06-06, catalog-only)
- skiller ≈ 2.1 GB. Public split: skills 42,763/794 · job_roles 22,315/2,381 · specializations 1,442/154 · categories 22/42.
- `skill_embeddings` 692 MB total but heap only 3.3 MB → ~689 MB is the pgvector index ⇒ **rebuild on replay**.
- embeddings carry no `organization_id` → scope via parent (`skill_id`/`job_role_id`).
