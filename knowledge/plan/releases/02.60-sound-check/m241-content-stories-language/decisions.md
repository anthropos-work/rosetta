## Adversarial review (close Phase 2c — 2026-07-22)

The scenario, not the fix, is recorded here so a future reviewer sees what was considered. Both were verified to be handled by the shipped code + existing coverage; no fix was required.

- **AR-1 — the two-point `LangToggle` derivation under a projection DROP.** `LangToggle` is set on each row from the **fixture-level** `bilingualTuples(set)` (over all fixture sessions, `content_manifest.go:372`), but `ValidateLanguageConsistency` re-derives tuple coverage from the **projected** rows (post-`BuildContentProducts`, which DROPS sessions that can't form a link — e.g. a missing `sim_slug`). Scenario: a bilingual tuple loses one language variant to a drop, so the surviving row still carries `LangToggle=true` (fixture-derived) while projected coverage is now solo — a toggle that would swap to a session that isn't in the manifest. **Verified handled (fail-closed):** `WriteContentManifest` runs the gate on the POST-drop projection (`content_manifest.go:532`→`:535`), and that exact input shape (a solo row marked `LangToggle=true`) is already a passing teeth-test branch (`TestValidateLanguageConsistency`, `content_manifest_test.go:128`) — the export FAILS rather than shipping a dangling toggle. No dangling toggle can reach a demo.
- **AR-2 — the cockpit's hard-coded English default vs the toggle's adaptive default.** `_content_session_row`'s server-side `lhide` uses the constant `_DEFAULT_CONTENT_LANG = "english"`, while `_content_lang_toggle` falls back to `langs[0]` when English is absent. If the toggle-able set ever lacked English while still having ≥2 languages, the button would highlight one language but every row would render hidden until a click. **Verified unreachable by design:** `validLanguages` is the closed set `{english, italian}` and `bilingualTuples` requires BOTH → the toggle-able set is always exactly `[english, italian]` whenever it has ≥2 entries, so English is always present and always the default (`TestBilingualTuples` pins "bilingual ⟹ both languages present"). The latent coupling is noted for any future 3rd-language extension: adding a language to `validLanguages` MUST re-derive the cockpit row-default from the actual toggle-able set, not the `english` constant.
