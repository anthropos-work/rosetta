# M202 Decisions

Implementation decisions with rationale (recorded during build). Design-time decisions live in
[`overview.md`](overview.md) + the consolidated capability spec
[`knowledge/plan/spec-drafts/playthroughs/spec.md`](../../../spec-drafts/playthroughs/spec.md) (v0.3).

| ID | Decision | Rationale | Date |
|----|----------|-----------|------|
| M202-D1 | The `playthroughs` rext section is a NEW top-level section with a MIXED toolchain: Go (`manifest/` model+validator, `report/` 4-state map, `cmd/ptvalidate` + `cmd/ptreport`) for the manifest/validator/report (matching the seeding module's `datadna`/`stackseed` CLI family + the datadna-gated requirement) + TypeScript (`e2e/`) for the Playwright page-object layer + specs. | The two halves reuse two different foundations — the Go seeding machinery (`--reset`, `datadna`) and the TS M42 e2e foundation (`cockpit-login`). One section, two languages, each matching its reuse target. | 2026-07-01 |
| M202-D2 | The validator does NOT import `stack-seeding` as a Go dep — it invokes `datadna measure-closure` as a SUBPROCESS. Preconditions resolve against a section-owned `seed/seed-worlds.yaml` INDEX (single-sourced with the dedicated seed), not by linking the seeding blueprint. | rext sections are self-contained modules with zero cross-section Go imports (the established convention). The datadna gate is a decoupled offset-port CLI (same as `stackseed`), so subprocess-invocation preserves the module boundary. | 2026-07-01 |
| M202-D3 | The dedicated seed reuses the seeding machinery UNCHANGED (a `stack.stories.yaml` consumed by `stackseed`), with its own two private orgs (Meridian Labs + Halcyon Retail) distinct from the demo showcase orgs. | Test data ≠ demo data (spec §5.4) + zero-fork of the seeding tooling. The seeding validator + roster-export accept it verbatim; the datadna closure gate applies. | 2026-07-01 |
| M202-D4 | LAYERING FINDING (proof run): seeding pt-world onto an ALREADY-SEEDED demo-1 collided — the stories model forces the FIRST story onto `LegacyOrgID` (22222222-…, the Clerkenstein default org), which on a seeded demo IS the showcase's default org (Cervato). So pt-org-a merged into Cervato → the `personas`/`hero-activity` seeders duplicate-keyed on Cervato's pre-existing `user_skills`/`user_bookmarks` (a DIFFERENT unique constraint than the idempotent-on-id COPY). FIX: pt-world carries a leading ANCHOR story (size 0, no heroes) that harmlessly re-declares the demo default org, pushing the REAL pt orgs to story index ≥1 so they get their own deterministic `StoryOrgID`s and never collide. Recorded as a genuine seeding-machinery constraint for a second-world-on-a-shared-stack seed (the demo default-org slot is single-tenant); the anchor-story is the zero-platform-edit, zero-fork workaround. | The `i>0 → StoryOrgID` rule in `blueprint/stories.go::EffectiveStories` hardcodes the first story onto LegacyOrgID with no opt-out field. Editing that is a seeding-machinery change out of M202's tooling-only scope; the anchor-story achieves the same effect declaratively. | 2026-07-01 |
