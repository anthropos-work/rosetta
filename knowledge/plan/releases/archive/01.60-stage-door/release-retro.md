# Release Retro: v1.6 "stage door"

**Shipped:** 2026-06-14, tag `v1.6`. **Milestones:** M27–M30. **Thesis:** turn "every operator hand-copies `platform/.env` into each repo, hoping no `.env` is missed or wrong" into "one values-blind engine ingests a secret source and provisions every repo's `.env`, scored against a secret-coverage DNA and gated non-fatally into bring-up."

## Headline
v1.6 delivered the secret-provisioning capability end-to-end — a new stdlib-only `stack-secrets` ext section: a **values-blind secret-coverage DNA** (55 genes / 6 repos), a DNA-driven dir/zip source reader that *structurally cannot* over-ingest, a `provision` engine that WRITES every repo's `.env` from the source, a **demo-aware `check`** wired non-fatally into `/dev-up` + `/demo-up` pre-flight, and the `/stack-secrets` skill — and **proved it observably on the real box** in the closing field-bake. A fresh demo-3 came up from a stack-dev-assembled `.agentspace/secrets`: provision wrote 26 / blanked 2, `check` scored Critical **100%**, **17 containers** reached UP. The defining safety moment was **defense-in-depth on the prod `DIRECTUS_TOKEN` holding under live verification**: provision write-BLANKs the strip family on the non-prod target AND the injection override strips it at compose-emit, and the live check confirmed the len-32 prod token was armed in **ZERO** containers — the fix16/17/M28 non-rearm class survived a real bring-up. The whole release moved secret bytes through exactly one function and **never leaked a value** into any output, log, or commit.

## Incidents this cycle
Every P0/P1/P2 incident across the release, with fate:
- **P2 (M28 harden Pass 2) — `preflight.sh` crashed mid-run under `set -u` on bash 3.2.** The non-demo path expanded an empty `demo_flag` array bare (`"${demo_flag[@]}"`), tripping an "unbound variable" abort on the macOS system bash (`#!/usr/bin/env bash` → 3.2). A non-demo `/dev-up` secret pre-flight would **CRASH instead of run** — the exact silent-break the non-fatal contract exists to prevent. Fixed Fate-1 with the conditional-expansion guard `${arr[@]+"${arr[@]}"}` + a `/bin/bash`-3.x regression; the shell-portability invariant was backfilled into `safety.md §2.8`. (ext `4a30ad4`)
- **P2 (M28 close review) — the demo secret pre-flight block sat ABOVE the lib-only test seam.** It was positioned in the `up-injected.sh` body above the `UP_INJECTED_LIB_ONLY` early-return, so sourcing the script lib-only (the `test_frontend_build.py` unit tests, which stub docker in a sandbox lacking the sibling `preflight.sh`) fired the bring-up pre-flight at source time → **20 frontend-build tests failed**. A bring-up ACTION had been placed where a function definition belongs. Fixed Fate-1 by moving it below the seam alongside the M19 VM pre-flight + a static positional regression. (ext `9742126`)
- **P1 (M30 field-bake) — the demo bring-up never PROVISIONED + the demo-aware gate silently SKIPPED.** Two defects in one: the demo bring-up only *checked* coverage (it ran from the operator's dev env, never moving bytes), and `preflight.sh` resolved its source path one level too shallow (doubled `.agentspace/.agentspace/secrets`) so the demo-aware gate **always silently skipped, exit 2** — vacuously green. Fixed Fate-1 by adding a non-fatal default-on provision step (`DEMO_NO_PROVISION=1` opt-out) that writes stack-demo's per-repo `.env` from the source + repoints the run's base env, and correcting the path to `EXT_ROOT/../..`. (tag `stage-door-m30` @ `29c922b`)
- **P1 (M30 field-bake) — `sentinel/DB_CONNECTION` falsely failed the gate at Critical 84.6%.** It was modeled critical/required, but is compose-injected config (hardcoded `environment:` entry, always overrides `env_file`; sentinel never reads it from a `.env`, and no `sentinel/.env` exists on stack-dev). Reclassified `waived-config` + a regression assertion — the textbook case the DNA's waived class exists for. (Fate-1, `stage-door-m30`)
- **P2 (M28 build, caught during build) — M28-D1 LiveKit de-alias.** The M27 DNA had aliased a key+secret *pair* as one value; provision's alias-fallback would have written the API key's value under the secret's key if the secret were missing — a silent wrong-value provision. De-aliased during build, reserving the alias mechanism strictly for true same-value families.
- **P2 (M27 build, caught during build) — 10 cross-repo secret-map omissions.** The build's `diff`-vs-stack-dev gate (the two-tier keep-listed gate, M27-D2) caught 10 real DNA omissions where a tracked secret was missing a gene → vacuously-green coverage. All 10 fixed Fate-1 during the build. (The naive "every declared key must be a gene" gate had first produced 111 false positives from config noise; scoping the gate-fatal class to the DNA's own tracked-secret universe made it honest AND usable.)

The roadmap had drafted the release as M26→M29, colliding on M26. ## Cross-milestone patterns
- **The "silently-skipped / vacuously-green gate" theme recurred — and was the release's most valuable catch class.** It appeared as a *design risk the gate was built to prevent* (a tracked secret missing a gene → coverage scores 100% over nothing — the anti-vacuous-100 guard + the M27-D2 two-tier gate exist for exactly this; the build caught 10 such omissions) and then as a *live defect of the gate's own wiring* (M30: the demo pre-flight's doubled source path made the gate exit 2 / silently skip — it scored nothing and called it pass). The lesson is uniform: a gate that can *skip* is as dangerous as one that's wrong, and only an end-to-end **live** run surfaces the skip — a dry-run scored green throughout.
- **The two-repo close-retry / two-pass pattern, navigated cleanly each time.** The code↔docs split (code → `rosetta-extensions`, tagged `stage-door-mNN`; docs/plan → rosetta on the milestone branch) held all release. M27's close was a two-repo close (ext doc-hygiene fix in `537aeff` + the rosetta record); M29 was rosetta-only (ext untouched on `main` @ `9742126` = tag `stage-door-m28`, no branch/tag/commit ext-side); M30 was two-repo (rosetta record + corpus reconciliation; ext field fixes @ tag `stage-door-m30`). Every milestone navigated the split without confusion.
- **A live field-bake earns its keep — clean-room tests structurally cannot see what it sees.** M30's two field bugs (never-provisioned demo; silently-skipping pre-flight) are precisely the "scored green but never moved the bytes / never ran the gate" class that a dry-run cannot expose, mirroring v1.5 M25's 4 field catches. Both fixed Fate-1 during the bake — they are the milestone's *deliverable*, not close-cycle incidents.
- **Values-blind scaled from read to write to live.** The invariant began as a read-side property (M27's source reader, only `ClassifyShape` permitted to touch a value as a discarded local) and survived the write path (M28 provision necessarily MOVES secret bytes, yet a reflection-walk safety test proves the only landing site is the gitignored target) and the live bake (assembly by `cp`/line-append only; provision stdout is key NAMES + counts; `.agentspace/secrets` stayed gitignored — verified). One value-carrying seam, `provision/io.go::sourceValues`, all release.
- **Inherited-deferral handling was strong, not just tracked.** Both within-release items resolved (DEF-M27-01 encrypted-zip DROPPED as a documented v1 scope boundary — a crypto dep would contradict the stdlib-only values-blind design, no live consumer; DEF-M27-02 per-gene profile-tag Fate-2 DISCHARGED at M28 close, the conditional never triggered). The DIRECTUS_TOKEN handoff M27→M28 — named blocks-release with a regression-test requirement at M27 close — landed cleanly. No repeat-defer pattern across all 4 close-audits + the release re-audit.
