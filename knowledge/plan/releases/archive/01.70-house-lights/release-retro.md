# Release Retro: v1.7 "house lights"

**Shipped:** 2026-06-15, tag `v1.7`. **Milestones:** M31–M32. **Thesis:** turn "a fresh browser at a demo's UI tier renders a blank page / a dead-port redirect, and the operator has to hand-fiddle to understand why" into "a fresh browser at a demo just renders the working UI, with ZERO manual fiddling" — the demo-UI-hardening release. Bring the house lights up so the show is actually visible.

## Headline
v1.7 fixed the two live demo-UI defects that the v1.6 field-bake's browser tier exposed, both with a small, contained, non-fatal change and zero platform-repo edits. **M31** automated a locally-trusted **mkcert** FAPI TLS cert into the demo bring-up — ONE branch in `up-injected.sh` step 3a-bis inside the existing keep-existing-cert guard (`command -v mkcert && [ "$NO_MKCERT" != 1 ]`), the historical openssl self-signed path factored into a single `gen_openssl_fapi_cert()` (1 defn / 2 call-sites, no two-copy drift), a `DEMO_NO_MKCERT=1` opt-out mirroring the `DEMO_NO_*` family — so clerk-js's dev-browser handshake to the fake FAPI (`https://127.0.0.1:35400`) stops hitting `net::ERR_CERT_AUTHORITY_INVALID` and next-web stops blanking. The leverage came from a **verified** no-touch guarantee: the 3 cert-CONSUMING files (`fake-fapi/main.go`, the `gen_injected_override.py` mount, `inject.py`) reference the cert by path only, so swapping which tool mints it changed nothing downstream (M31-D4, confirmed at the Phase-0b audit, not assumed). **M32** fixed the sibling studio-desk dead-`:9100` redirect: a three-layer Docker-env precedence chain (verified, not assumed) had the base platform compose's `NODE_ENV=development`/`FRONTEND_PORT=9100` win the demo override's *additive* env merge over the image's baked `ENV NODE_ENV=production` → `isProduction=false` → `res.redirect('http://localhost:9100/home')`, a dead port. The 1-line-class fix pins `NODE_ENV=production` (+ `FRONTEND_PORT=9000` belt-and-suspenders) into the studio-desk override's env dict, winning the additive merge back to the single-port production `sendFile` path; the dead un-offset `:9100` CORS origin was dropped and the `:9100`→single-port-`9000` doc/CORS sweep ran across `frontend-tier.md` + the demo-up SKILL. The whole release moved no cert/key bytes to stdout/log (values-blind), kept every bring-up step non-fatal, and touched zero `.go` files. **Release verdict GREEN** — all 9 close-release sweeps GREEN, zero blocking, zero should-fix code/doc.

## Incidents this cycle
Every P0/P1/P2 incident across the release, with fate. **No production / field P0 or P1 fired in v1.7** — the two defects the release exists to fix were *already* live before it opened (they are the deliverables, not cycle incidents). What surfaced during the cycle were test-harness / latent-test defects, each a P2-class catch at build or harden:
- **P2 (M31 harden) — the false-greening cert test harness (`set -uo` vs `set -euo pipefail`).** The build's extracted-block test harnesses sourced the cert block under `set -uo pipefail`, but the real bring-up runs `set -euo pipefail` (`up-injected.sh:13`). Without `-e`, the cert step's `mkcert -install … || true` swallow is inert, so the install-failure harden test would have **false-greened** — passed even with `|| true` deleted. Fixed Fate-1 by switching all three `FapiCertStep` harnesses to `set -euo pipefail`; **mutation-proven** the test now discriminates (removing `|| true` → `CalledProcessError`). `up-injected.sh` was byte-identical before and after — the production code was already correct; only the test harness needed the strict-mode fidelity. (M31-D6)
- **P2 (M32 build) — two latent env-masked YAML tests surfaced under PyYAML.** Both pre-existing on the `house-lights-m31` tag (M32-D3): (1) `test_frontend_blocks_parse_to_valid_compose` asserted studio-desk ports `["29000:9000","29100:9100"]` while the generator only ever emitted the single `["29000:9000"]`; (2) `test_a_plain_service_parses_to_ports_only` predated the universal fix16/17 prod-`DIRECTUS_TOKEN=` strip. Both were masked because the YAML-structural tests `@skipIf(yaml is None)` and PyYAML is absent in the default `python3` env — they only fail when PyYAML IS present. Surfaced by running the suite under BOTH interpreters (py3.11+PyYAML and py3.14-no-PyYAML) at build; fixed Fate-1.
- **P2 (M32 harden) — the CORS over-removal blind spot.** The CORS test had only membership asserts (`19000` present, `19100` absent) and did NOT pin the full surviving origin set — so over-removing the kept `3001` next-web origin passed the entire 88-test suite (mutation-confirmed). Closed Fate-1 with a new exact-set assertion (`origins == [3000,3001,9000]+offset` in emit order, commit `7b17c39`) that catches over-removal of any kept origin AND an accidental re-add of `9100`.

Close-time findings, all Fate-1, no functional gap:
- **M31 close — 2 findings.** (1) `demo-stack/README.md` quoted "13 unit tests" (×2, stale since ~M16/M28; actual 50) → reconciled to 50 + a note distinguishing it from GUIDE's guarded "28" curated-subset count (the Phase-4 handbook-count-reconciliation discipline catching pre-existing drift at the milestone boundary, M31-D8); (2) an adversarial scenario recorded in `decisions.md` (a zero-byte/truncated cert + the existence-only keep-guard → NOT a regression, identical to pre-M31 behavior, documented repair path `rm fapi.crt` + re-up).
- **M32 close — 4 findings.** 3 decision-tag blends (D1/D2/D4 reference-tagged into `frontend-tier.md`'s single-port + CORS notes) + 1 adversarial-scenario record (the additive-merge keeping base `NODE_ENV=development`, defended live via a merge-probe). No code changed beyond the doc tags — the milestone shipped clean from build + harden.

## Cross-milestone patterns
- **"Close-verify-by-composition because no demo is up" recurred — cleanly — in BOTH milestones.** demo-3 had been torn down before either close, so neither M31 nor M32 re-spun a fresh `/demo-up`; each instead decomposed its observable-behavior gate into the one load-bearing link that was actually failing + the already-proven links, and proved that link. M31-D7: chromium *trusts* the mkcert cert (default context, no `ignoreHTTPSErrors`: 200 / no cert error) vs chromium *rejects* the old openssl self-signed (`ERR_CERT_AUTHORITY_INVALID` — the exact blank-page cause) + cert-trusted→renders + the 11 `FapiCertStep` tests. M32-D5: the production pin is set (regression test, mutation-checked 4 ways) → `isProduction=true` → the production block serves every dev-block route via `sendFile`+`express.static`+SPA fallback (code-read M32-D1, "NO GAP") → no dead-`:9100` 302, no 404; *strengthened at close* with a live merge-probe (the override built, parsed via the repo's `!override`-aware loader, the additive list-merge against base `[development, 9100, PORT=9000]` simulated → the `production`/`9000` pins win, last-`VAR=`-wins, exactly one `environment:` key, no `9100`). Both rationales hold as necessary+sufficient. **The honest residual:** a single fresh `/demo-up` would give the live end-to-end render proof for BOTH fixes at once (the same fresh demo-N's next-web would exercise M31's trusted FAPI cert and its studio-desk would exercise M32's single-port pin) — composition proved the load-bearing links, but the one combined live render was deferred with the demo torn down. The verify-by-composition contract is genuine resolved-work closure here (each cites merged regression tests confirmed present at ext HEAD), not a disguised punt.
- **"Harden catches a false-greening / over-permissive test" recurred across M31 AND M32 — the release's most valuable catch class.** M31's harden caught a *false-greening* harness (the `set -uo` vs `-euo` strict-mode gap let the `|| true` swallow go untested — a test that passed for the wrong reason). M32's build+harden caught *over-permissive* tests twice: the two env-masked YAML asserts (green-by-skip since before M31 because PyYAML was absent) and the membership-only CORS test (green even when a kept origin was over-removed). The uniform lesson: a test that passes when it shouldn't is as dangerous as one that's wrong, and only adversarial pressure — mirroring the real script's `set` flags, running under every supported interpreter, pinning the exact set instead of membership — surfaces it. M31-D6's strict-mode-harness rule (a harness extracting-and-running a bring-up block must mirror the real script's `set` flags) generalizes beyond this release.
- **Code-read-before-trust on the load-bearing question, both times.** Each milestone had exactly one question that could have shipped a user-visible break, and each was settled by reading the real cloned source before acting — not by "the smoke will tell us." M31: are the 3 cert-consumers path-only? (Phase-0b audit: yes, before the build started.) M32: does the production block cover every route the dev block redirected to? (read `src/index.ts` 148–272: yes, NO GAP, before flipping the flag.) A small, low-risk change by construction in both cases.
- **The two-repo code↔docs split held all release.** Code → `rosetta-extensions`, tagged `house-lights-mNN`; docs/plan → rosetta on the milestone branch. Both milestones were genuine two-repo closes (rosetta record + corpus doc rewrite/sweep; ext field code at the tag) and each navigated the split without confusion. The release diff confirms the discipline: rosetta `main...release/01.70-house-lights` = 23 files, +1527/-72, **docs + planning ONLY** (0 `.py`/`.go`/`.sh`/`.ts`); ext `868a68a..7b17c39` = exactly the 5 claimed files, +356/-26, all within `demo-stack/` + `stack-injection/` (0 `.go`, 0 platform-repo paths).
- **Small, well-scoped surface end-to-end.** v1.7 is the smallest release of the family by diff: one bash branch + one factored helper (M31), a few env-dict pins + a CORS-set edit (M32), and a doc sweep. The harden + close passes each found exactly one or two genuine gaps and a re-scan surfaced nothing further — the kind of release where the rigor is in the verification, not the volume.
