# M40 Decisions

Implementation decisions with rationale (recorded during build). Design-time context lives in the spec
([`.agentspace/profile_gaps.md`](../../../../.agentspace/profile_gaps.md); root-cause workflow w7t4wq2z4).

| ID | Decision | Rationale | Date |
|----|----------|-----------|------|
| M40-D1 | **The cms anonymous read path is genuinely token-less.** `DIRECTUS_TOKEN` is EMPTY on the per-stack cms container (`docker exec demo-3-cms-1` → `TOKEN_SET=`), and the Directus client only sets `Authorization: Bearer` when `c.token != ""`. So the **public policy** (`abf8a154-…`) is the operative permission set for every library/activity read — synthesizing public-policy rows IS the correct, sufficient lever (no platform change, no service-token path). | Refutes the build-sub-agent's initial investigation hypothesis that a static service token bypasses the public policy. Verified live on demo-3. | 2026-06-24 |
| M40-D2 | **Root cause is bigger than the original spec framing: the per-stack Directus has `directus_relations = 0` and `directus_fields = 0`.** Only the 9 `directus_collections` + the 5 prod-public `directus_permissions` rows are synthesized. Without the relational metadata, the O2M/M2M *aliases* (`simulations.sequences`, `simulations.library_categories`, `library_categories.macro_category`, `sequences.roles`, `task_checks.sub_checks`) are **unknown to Directus** → an explicit `fields=sequences` returns `"You don't have permission to access field 'sequences' … or it does not exist"`, and a `fields=sequences.scenario_intro` is **silently stripped**. The earlier "O2M stripped under the public policy" symptom is this missing-metadata, NOT a Directus policy limitation. **(c) does NOT need a platform nil-guard** — it needs the relations/fields registered. | Verified live: `sequences|read` IS already granted on demo-3's public policy and `sequences` serves perfectly when read DIRECTLY (`/items/sequences` returns full `scenario_intro`); only the NESTED read from `simulations` fails, and `directus_relations`=0 explains it exactly. This collapses the "library half ships, activity half escalates for platform sign-off" fork — BOTH halves ship in tooling. | 2026-06-24 |
| M40-D3 | **Synthesize the full relational web dynamically (relations + fields + the missing target/junction collections + permissions), not a hand-enumerated list.** Capture every `directus_relations`/`directus_fields` row whose endpoints lie within the closure of the served content collections + the collections the cms field tree expands (`library_categories`, `library_macro_categories`, the `simulations_library_categories`/`skill_paths_library_categories` junctions), then grant public read on the newly-served collections + on `versions` (directus_versions, system collection). | Matches structure.go's existing dynamic/version-robust capture philosophy (no hardcoded collection list). Both `directus_relations` and `directus_fields` carry NONE of the firewall `TenantScopeColumns` (verified via sanctioned prod structural read) → admissible under the existing `AssertStructuralMetadata` carve-out, extended not loosened. | 2026-06-24 |
| M40-D4 | **The closure must require BOTH relation endpoints on-stack — an alias to an off-stack target is DROPPED, not merely ungranted; and EVERY relational `special` (o2m/m2m/m2a/files/m2o/file) is gated on its surviving relation.** A no-backing-column alias (o2m/m2m/m2a/files) registered without its relation → Directus emits `SELECT <coll>.<alias>` vs a missing column → 500 (the `sequences.assets_files` class, off-stack `sequences_files` junction). A real-FK relation (m2o/file) whose relation is dropped → Directus returns the bare FK STRING → cms unmarshal failure (the `job_position` class). Non-relational fields (cast/date/user/uuid/group) are always kept. | Two distinct live failure modes (verified on demo-3: `column sequences.assets_files does not exist`; `cannot unmarshal string into JobSimulation.data.job_position`). The closure is the precise both-endpoints predicate that keeps exactly the cms read path's needs and drops everything off-stack. | 2026-06-24 |
| M40-D5 | **Add `resource` + `job_position` to `servedCollections` (their tables already exist via the dynamic structure DDL); synthesize a public-read grant for EVERY served collection prod doesn't grant.** `servePermissionsRowsSQL` only COPIES prod's 5 public collections, but cms's library list expands M2O targets `skill_paths.video → resource` and `simulations.job_position → job_position` — an ungranted target makes Directus return the bare FK string (unmarshalable). With no replayed rows the M2O expansion is NULL (cms tolerates pointer/omitempty). All `job_position`/`resource` referenced rows are public (`tenant_id` NULL), so the no-filter synth grant stays within the public boundary. | The copied-only grant path was the gap. The synth grant is `WHERE NOT EXISTS`-guarded so it never overrides the copied `simulations`/`skill_paths` rows (which keep their `status='published'` filter). | 2026-06-24 |
| M40-D6 | **Grant `directus_versions` BOTH read AND create on the public policy (collection name `directus_versions`, NOT the `versions` API path).** cms `GetLatestOrCreateVersion` reads `/versions` (403 fatal) and CREATEs a version when none exists. Prod's `skill_paths` versioning has ~one version per published path so cms reads; the per-stack table is empty (not replayed) so cms CREATEs — granting create lets cms self-heal instead of replaying prod's 539 version rows + deltas. Verified live: collection `'versions'` → still 403; `'directus_versions'` → 200; +create → full skill-paths library serves. | The CREATE path is the per-stack-empty consequence of not replaying directus_versions rows; self-heal-via-create is far cheaper + content-equivalent vs a 539-row + delta replay. | 2026-06-24 |
| M40-D7 | **KPI "AI simulations completed" = 0 is OUT of M40 scope (Fate-2 → M42e/M42m).** Its source `public.local_jobsimulation_sessions` has no CMS dependency (the feed fix is a Directus-serve gap; the KPI reads jobsimulation directly), so it is a separate frontend/auth-context concern. The M42e (employee) + M42m (manager) coverage milestones already own end-to-end per-vantage page rendering, which is where a residual KPI-zero would surface and be diagnosed. Not coupled to the feed fix. | overview.md Open question asked to re-verify + flag separately if it persists. It is genuinely a different surface (frontend KPI vs CMS serve); the right home is the coverage sweep that exercises every page, not this serve-grant. | 2026-06-24 |
