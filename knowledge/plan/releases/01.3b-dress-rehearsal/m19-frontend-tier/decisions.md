# M19 — Decisions

_Implementation decisions with rationale. ID scheme: M19-D1, M19-D2, … Open questions from `overview.md` get resolved here during build._

| ID | Decision | Rationale | Date |
|----|----------|-----------|------|
| M19-D1 | The frontends are **appended** to the override (like the fake servers), not walked out of `cfg`. | next-web/studio-desk live in the `frontend`/`studio-desk`/`all` profiles, NOT the demo's `graphql` profile, so they never appear in the resolved compose `cfg`. `profiles: !override [graphql]` makes them join the demo's existing `--profile graphql up`. | 2026-06-09 |
| M19-D2 | Removed the stale `next-web-app` `REUSE_DEV` entry (KB-1). | It never emitted (next-web isn't in the demo graphql profile) and M19 supersedes it with a per-demo BUILT image (`demo-N-next-web`), not a dev-image reuse — keeping both would be a contradictory double-path. | 2026-06-09 |
| M19-D3 | next-web pk via gitignored `apps/web/.env.local`; studio-desk pk as a direct build-arg. | Verified against the real Dockerfiles: next-web's `Dockerfile.dev` declares ARGs only for the 3 URLs (no pk ARG) → `.env.local` is the no-edit path `next build` reads; studio-desk's `Dockerfile.dev` DOES declare `VITE_CLERK_PUBLISHABLE_KEY` as an ARG → pass it directly. Both keep the platform repo byte-pristine. | 2026-06-09 |
| M19-D4 | The next-web `.dockerignore` is tooling-owned + applied **transiently** (non-clobber, trap-removed); studio-desk's own is left untouched. | next-web ships no `.dockerignore` (2.8 GB context); a transient untracked one keeps the repo git-clean. studio-desk already ships one excluding `node_modules`/`dist`/`.git` — editing it would be a platform edit, and it's unnecessary. | 2026-06-09 |
| M19-D5 | The 12 GB VM assert + the frontend-build failure are **non-fatal** (warn, never abort). | Runtime is cheap (~0.66 GiB both stacks); only the build spikes. A soft RAM heuristic or a transient build failure must never block an otherwise-good bring-up — mirrors M18's non-fatal verify + M17's `set -e` race hardening. `DEMO_VM_MIN_GIB`/`DEMO_NO_UI` are the operator escapes. | 2026-06-09 |
| M19-D6 | ant-academy launched **natively** by the tool, Clerk-free via its own `BENCHMARK_VISUAL_BYPASS`, with a documented manual fallback. | Resolves the overview's open question (launch vs document) toward "launch it, fall back if fiddly". The academy is Vercel-native (not dockerizable cleanly) and Clerk-only; `BENCHMARK_VISUAL_BYPASS` gives anonymous Clerk-free browse with zero academy-repo edits. The fallback covers the absent-clone / no-deps (Font Awesome token) / old-Node cases without aborting. | 2026-06-09 |
| M19-D7 | The verify net scopes the frontends **iff UI is on** (registry has them; the bring-up passes `--services` accordingly). | Adding next-web/studio-desk to `stack-verify`'s SERVICES registry lets a UI-on demo verify them; passing the scoped `--services` (frontends only when `NO_UI != 1`) stops a `--no-ui` demo false-`down`ing an absent frontend. Fills the hook M18 explicitly left for M19. | 2026-06-09 |
| M19-D8 | `HERE` resolves via `${BASH_SOURCE[0]}` (not `$0`); add a `UP_INJECTED_LIB_ONLY` source seam. | `$0` is the sourcing shell under `source`, breaking path resolution; `BASH_SOURCE[0]` is correct under both exec and source. The lib-only seam lets the frontend-build functions be unit-tested with a stubbed `docker` without running the bring-up. | 2026-06-09 |
| M19-D9 | The native academy launch detaches **all three** stdio fds (`</dev/null >>log 2>&1` on the whole launch subshell). | A backgrounded daemon that keeps the caller's stdout/stderr pipe open hangs any `capture_output` caller (and the test harness) until the daemon exits — the classic background-job pipe-hang. Redirecting the subshell's 0/1/2 (not just the inner npm) releases the pipe. Surfaced + fixed during §4 testing. | 2026-06-09 |
