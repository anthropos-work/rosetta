## Adversarial review (close Phase 2c)

The harden pass already pinned the milestone's load-bearing invariant adversarially ‚Äî `TestZeroPlatformRepoEdit` stands up a **real git repo** as the build context and asserts `git status` stays empty after the (stubbed) build, including the **failed/aborted-build** path (the `RETURN`-scoped trap), plus a `git check-ignore` fence that the pk overlay path can never be tracked even mid-build. The close re-examined three further scenarios against the shipped code; all are already handled:

- **Scenario: the `RETURN` trap in `build_frontend_next_web` clobbers a trap the caller relied on, or leaks between the two serial frontend builds.** Handled ‚Äî the trap is set inside the function and `trap - RETURN` clears it on return, so it fires once (cleaning next-web's overlay + transient `.dockerignore`) before `build_frontend_studio_desk` runs. studio-desk sets no trap (its own `.dockerignore` is untouched, pk is a direct ARG). No cross-build leak. Mutation-pinned by the harden tests (dropping the cleanup fails the guard).
- **Scenario: a stale academy pidfile from a prior failed launch makes a re-up think the academy is already running.** Handled ‚Äî `ant-academy.sh` gates the "already running" branch on `kill -0 "$(cat PIDFILE)"`, so a dead pid falls through to a fresh launch; `--stop` likewise no-ops on a dead pid. Harmless by design (recorded in progress.md Stop condition) ‚Äî no fix warranted.
- **Scenario: `platform/.env` absent ‚Üí the pk/Clerk-key bake produces an invalid overlay.** Handled ‚Äî both the next-web `.env.local` overlay and the academy `.env.local` treat the Clerk keys as **optional** (the academy browses anonymously under `BENCHMARK_VISUAL_BYPASS`; next-web's pk comes from `inject.py`, not `platform/.env`). Covered by `test_env_overlay_written_without_clerk_keys_when_platform_env_absent`.

No new adversarial finding required code changes ‚Äî the invariants held.
